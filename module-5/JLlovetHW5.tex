\noindent\textbf{Problem 1:}

A deque (pronounced deck) is an ordered set of items from which items may be deleted at either end and into which items may be inserted at either end. Call the two ends left and right. This is an access-restricted structure since no insertions or deletions can happen other than at the ends. Implement the deque as a doubly-linked list (not circular, no header). Write InsertLeft and DeleteRight.

\medskip
\noindent\textbf{Answer:}
\medskip

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Go}
type Node {
    data: T
    previous: Node | null
    next: Node | null
}

type Deque {
    left: Node | null
    right: Node | null
}

function InsertLeft(d: Deque, data: T) {
    node = Node(data)
    if d.left == null {
      d.left = node
      d.right = node
    }
    else {
      node.next = d.left
      d.left.previous = node
      d.left = node
    }
    return d
}

function DeleteRight(d: Deque) {
    if d.right == null {
      return d
    }
    if d.right.previous == null {
      d.right = null
      d.left = null
      return d
    }
    if d.right.previous != null {
      // have to store d.right in a variable to avoid prematurely removing reference to it  
      currentRight = d.right
      d.right.previous.next = null
      d.right = currentRight.previous
      return d
    }
}
\end{minted}

\noindent\textbf{Problem 2:}

Implement a deque from problem 1 as a doubly-linked circular list with a header. Write InsertRight and DeleteLeft.


\medskip
\noindent\textbf{Answer:}
\medskip

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Go}
type Node {
    isHeader: boolean
    data: T
    previous: Node | null
    next: Node | null
}

type Deque {
    header: Node
    left: Node | null
    right: Node | null
}

function newDeque(){
    // To be called in hypothetical program to initialize
    // Handled idiomatically in Python/Java/Go/C++/etc constructors
    d = Deque()
    d.header = Node(data=0, isHeader=True)
    // header.data represents the length of the list excluding the header
    d.header.data = 0
    d.left = d.header
    d.right = d.header
}

function InsertRight(d: Deque, data: T) {
    node = Node(data=data, isHeader=False)
    // Case length == 0
    if d.header.data == 0 {
        node.previous = d.header
        node.next = d.header
        d.left = node
        d.right = node
        d.header.next = node
        d.header.previous = node
    }
    // Case length > 0
    if d.header.data > 0 {
        node.previous = d.right
        d.right.next = node
        node.next = d.header
        d.header.previous = node
        d.right = node
    }
    d.header.data += 1
    return d
}

function DeleteLeft(d: Deque) {
    // Case length == 0
    if d.header.data == 0 {
        return d
    }
    // Case length == 1
    if d.left.next.isHeader {
        d.left = d.header
        d.right = d.header
        d.header.next = d.header
        d.header.previous = d.header
    }
    // Case length > 1
    if not d.left.next.isHeader {
        d.left.next.previous = d.header
        d.header.next = d.left.next
        d.left = d.left.next
    }
    d.header.data -= 1
    return d
}
\end{minted}

\noindent\textbf{Problem 3:}

Write a set of routines for implementing several stacks and queues within a single array. Hint: Look at the lecture material on the hybrid implementation.

\medskip
\noindent\textbf{Answer:}
\medskip

We can take advantage of the shared characteristics of the ADTs for queues and stacks to make reasoning about this simpler. The ADTs for both data structures, at their core, each satisfy the following interface, just with different semantics and implementation details for the underlying behavior (e.g. LIFO vs FIFO).

\begin{tcolorbox}[colback=gray!10!white, colframe=gray!80!black, title=Note]
  In the following, consider a \texttt{Sequence} an abstraction over stacks, queues, arrays, linked lists, and other linear storage types. It's just a collection of elements with an ordering. Characteristics such as whether access is random or sequential, the sequence has a particular ordering, and so forth, are handled in an implementation or in the semantics of the ADT that consume it, rather than in the \texttt{Sequence} itself. The notation \texttt{Sequence<T>} signifies that a \texttt{Sequence} contains elements of type \texttt{T}. Convention from Java.
  \medskip

  Note that the notation \texttt{f: A -> B} $\equiv$ \texttt{f(A) -> B}, and \texttt{g: A -> B -> C} $\equiv$ \texttt{f(A,B) -> C}, signifying that a function \texttt{f} takes one argument of type \texttt{A} and returns a value of type \texttt{B}, and that a function \texttt{g} takes two arguments of types \texttt{A} and \texttt{B} and returns a value of type \texttt{C}. Convention from Haskell and category theory.
\end{tcolorbox}

We will be using a hybrid sequence management approach, where the data is in an underlying array for storage, and the sequence data structures we have consume that array will just keep track of the head if they are stacks or the head and tail if they are queues. Nodes will be as follows.

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Go}
type Node {
    data:     T
    next:     Node
    previous: Node
}
\end{minted}

The underlying storage mechanism is as follows:

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Go}
interface LinearStorage {
    ordering:   LIFO | FIFO
    data:       Sequence<T>
    head:       Node
    tail:       Node
    maxSize:    int
    size:       int
    isEmpty:    Sequence<T> -> bool
    isFull:     Sequence<T> -> bool
    peek:       Sequence<T> -> T | null
    contains:   Sequence<T> -> T -> bool
    addItem:    Sequence<T> -> T -> bool
    removeItem: Sequence<T> -> T | null
}
\end{minted}

For an object of type \texttt{Stack}, \texttt{addItem} is usually called \texttt{push}, and it satisfies \texttt{LIFO} semantics. Similarly, for an object of type \texttt{Queue}, \texttt{addItem} is usually called \texttt{enqueue} or \texttt{add}, and it satisfies \texttt{FIFO} semantics. So, we can develop a generic way to support both of these types for a given underlying array.

\medskip

We will create a \texttt{StorageManager} that has the following attributes, along with a few functions that consume it. Real implementations might store metadata about consumers, among other details that we will not consider here.

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Go}
type StorageManager {
    maxSize:        int
    freeSpace:      Stack(maxSize)
    data:           Sequence<T>
}
newStorageManager: int            -> StorageManager
isEmpty:           StorageManager -> bool
isFull:            StorageManager -> bool
addItem:           StorageManager -> LinearStorage -> T -> (bool, StorageManager, LinearStorage)
removeItem:        StorageManager -> LinearStorage -> (T | null, StorageManager, LinearStorage)
\end{minted}

Why the tuples of return values? Our functions rely on multiple items, and there are multiple items that are potentially modified during the calculation. We are returning all the values so that the caller can appropriately update their references, rather than rely on hidden mutation of state.

\pagebreak
\noindent Reframed in a more C-language-family style to illustrate the design:

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Python}
type StorageManager{
    maxSize: int
    freeSpace: Stack(maxSize)
    data: Sequence<T>
}

storage = StorageManager(10)
// constructor allocates 10 items of space
// then it fills the storage.freeSpace stack with positions 0-9
queue = LinearStorage(FIFO)
stack = LinearStorage(LIFO)
item = 3.14

isEmpty(storage) // true
isFull(storage) // false

addItem(storage, queue, item) // returns: (true, storage, queue)
// storage.freeSpace was popped, the value 3.14 is written to storage.data[0],
// the return values indicate that the addition was successful and provide
// the updated queue (with its first data element updated) and StorageManager

addItem(storage, stack, item) // returns: (true, storage, stack)
// storage.freeSpace was popped, the value 3.14 is written to storage.data[1],
// the return values indicate that the addition was successful and provide
// the updated stack (with its first data element updated) and StorageManager

removeItem(storage, queue)    // returns: (3.14, storage, queue)
// storage.freeSpace had the value 0 pushed onto it, since queue is
// no longer using it.
// The queue is updated to be empty, with its first position represented as -1.
\end{minted}

\noindent Presuming that storage were full, we would see the following behavior:

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Python}
addItem(storage, stack, item) // returns: (false, stack, storage)
// no modification to stack or storage, since the underlying storage is full
\end{minted}

Coming back to the StorageManager functions themselves, here is a possible approach:

\begin{minted}[linenos,breaklines,breakanywhere=true,fontsize={\fontsize{8pt}{9pt}\selectfont}]{Go}
type StorageManager {
    maxSize:        int
    freeSpace:      Stack(maxSize)
    data:           Sequence<T>
}

function newStorageManager(maxSize: int) -> StorageManager {
    s = StorageManager()
    s.maxSize = maxSize
    space = Stack(maxSize)
    for i in maxSize..0 {
        space.push(i)
    }
    s.freeSpace = space
    s.data = array[maxSize]
    return s
}

function isEmpty(s: StorageManager) -> bool {
    // checks internal stack used to track free locations
    return s.freeSpace.isEmpty()
}

function isFull(s: StorageManager)  -> bool {
    // checks internal stack used to track free locations
    return s.freeSpace.isFull()
}

function addItem(s: StorageManager, seq: LinearStorage, item: T) -> (bool, StorageManager, LinearStorage) {
    if s.isFull() {
        return (false, s, seq)
    }
    nodeLocation = s.freeSpace.pop()
    node = Node()
    node.data = item
    if seq.isEmpty() {
        node.previous = null
        node.next = null
        seq.head = nodeLocation
        if seq.ordering == FIFO { // queue
            seq.tail = nodeLocation
        }
    }
    if not seq.isEmpty() {
        if seq.ordering == LIFO { // stack
            node.next = seq.head
            node.previous = null
            s.data[nodeLocation] = node    
            s.data[seq.head].next = nodeLocation
            seq.head = nodeLocation
        }
        if seq.ordering == FIFO { // queue
            node.previous = seq.tail
            node.next = null
            s.data[nodeLocation] = node
            s.data[seq.tail].next = nodeLocation
            seq.tail = nodeLocation
        }
    }
    seq.size += 1
    return (true, s, seq)
}

function removeItem(s: StorageManager, seq: LinearStorage) -> (T | null, StorageManager, LinearStorage) {
    if s.isEmpty() or seq.isEmpty() {
      return (null, s, seq)
    }
    if seq.ordering == LIFO { // stack
        nodeLocation = seq.head
        node = s.data[nodeLocation]
        if seq.size == 1 {
            seq.head = null
        }
        else if seq.size > 1 {
            s.data[node.previous].previous = null
            seq.head = node.previous
        }
    }
    if seq.ordering == FIFO { // queue
        nodeLocation = seq.head
        node = s.data[nodeLocation]
        if seq.size == 1 {
            seq.head = null
            seq.tail = null
        }
        else if seq.size > 1 {
            s.data[node.next].previous = null
            seq.head = node.next
        }
    }
    seq.size -= 1
    s.freeSpace.push(nodeLocation)
    return (node.data, s, seq)
}
\end{minted}
